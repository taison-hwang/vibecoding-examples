# UI 요구사항 및 예외 처리(UX)

### 1. 메인 화면 (회의 요약 및 전송 탭)

* **회의 내용 입력 영역:** 일상 언어로 작성된 회의 텍스트를 붙여넣을 수 있는 대형 Textarea.
* **액션 버튼:** "AI 요약 및 액션 아이템 도출" 버튼.
* **인터랙션:** 클릭 시 버튼 내부에 로딩 스피너가 표시되며, 완료될 때까지 버튼은 비활성화(Disabled) 처리됩니다.


* **결과 출력 영역:** Gemini가 리턴한 JSON 데이터를 바탕으로 '개요' 섹션과 '액션 아이템 테이블(담당자, 업무, 기한)'로 시각적으로 정돈하여 렌더링합니다.
* **전송 버튼:** "등록된 팀원에게 이메일 전송" 버튼.
* **상태 제어:** 요약 결과가 없거나 수신자 이메일 목록이 비어있을 때는 버튼이 비활성화됩니다.



### 2. 에러 및 예외 처리 UX (중요 보완)

* **Gemini 변환 실패 시:** AI 응답 지연이나 파싱 에러 발생 시, 결과 영역에 에러 메시지("요약 도출에 실패했습니다. 다시 시도해주세요.")와 함께 [재시도] 버튼을 노출합니다.
* **이메일 전송 실패 시:** SMTP 인증 실패, 네트워크 오류, 잘못된 수신자 주소 등으로 인해 전송이 실패한 경우, 단순 성공 알림 대신 "일부 또는 전체 이메일 전송에 실패했습니다. 백엔드 설정을 확인하세요."라는 경고 모달/토스트 메시지를 띄우고 전송 버튼을 다시 활성화합니다.

### 3. 설정 화면 (팀원 이메일 관리 탭)

* **이메일 입력 폼:** 텍스트 인풋과 "추가" 버튼. 정규식을 통해 이메일 형식이 아닐 경우 "올바른 이메일 형식이 아닙니다"라는 에러 문구를 인풋 하단에 붉은색으로 출력합니다.
* **멤버 리스트 / 빈 상태:** 등록된 이메일이 태그 형태로 나열되며 [X] 버튼으로 삭제 가능합니다. 등록된 데이터가 없으면 "등록된 팀원이 없습니다."라는 안내 문구를 노출합니다.

# 기술 스택 및 보안 지침

### 1. ⚠️ 크리티컬 보안 지침 (API 키 노출 방지)

* **Gemini API 키는 절대로 프론트엔드(React/Vite) 코드나 환경변수(`VITE_`)에 포함시켜 브라우저에 노출해서는 안 됩니다.**
* API 키는 반드시 백엔드 서버의 `.env` 파일에만 안전하게 보관하고, 프론트엔드는 백엔드 엔드포인트(`/api/summarize`)를 통해서만 AI 기능을 호출하도록 아키텍처를 강제합니다.

### 2. 백엔드 및 이메일 기술 스택 조합

실습생의 혼선을 방지하기 위해 백엔드 언어에 따른 이메일 라이브러리 조합을 명확히 분리합니다. 실습 환경에 맞춰 한 가지 조합을 선택하여 진행합니다.

| 백엔드 선택 (택 1) | 이메일 연동 라이브러리 | 환경 변수 관리 |
| --- | --- | --- |
| **Node.js** (Express) | `Nodemailer` | `dotenv` |
| **Python** (FastAPI) | `smtplib` + `email.mime` | `python-dotenv` |

### 3. 이메일 발송 환경 전제 조건 및 대안

Gmail SMTP를 사용하여 메일을 발송할 때의 제약사항을 인지하고 환경을 준비합니다.

* **Gmail SMTP 사용 시:** 발송용 Google 계정에 **'2단계 인증'이 반드시 활성화**되어 있어야 하며, Google 계정 설정에서 '앱 비밀번호(App Password)'를 생성하여 백엔드 환경변수에 세팅해야 합니다.
* **인증 방식 대안:** 실습 계정의 보안 설정 변경이 어렵거나 Google 정책 변화로 앱 비밀번호 발급이 원활하지 않을 경우, 개발자용 무료 SMTP 테스트 서비스인 **Mailtrap**이나 **Resend**(무료 등급)를 대안으로 연동하여 실습을 진행합니다.

## 2. 핵심 원칙

- **"80% 품질"의 뜻을 분명히 한다.** 합의를 끌어내기 위한 수준이며, 핵심 흐름은 동작하고 시각 완성도는 실제에 가깝게, 단 성능·보안·예외 처리는 간이 구현으로 둡니다. 이 경계를 클라이언트에게 먼저 설명합니다.
- **기존 문서에서 출발한다.** 추측으로 화면을 짜지 않고, 요구사항·ADR·DB·API에서 화면/입출력/권한/흐름을 추출합니다. 문서에 없는 부분은 "(가정)"으로 표시하고 데모 때 확인합니다.
- **체험 가능한 형태로 만든다.** 정적 시안이 아니라 실제로 로그인하고 화면을 이동할 수 있는 형태로 만들어, 사용성 피드백을 받습니다.
- **반응형을 기본으로 한다.** 모바일·데스크톱을 같이 고려합니다.
- **실습 규모로 좁힌다.** 바이브코딩 실습용이므로 화면을 **2~3개 수준**으로 한정하고, 가짜 데이터로 시작하는 **MVP 중심**으로 만듭니다. 한 번의 실습으로 끝낼 수 있는 범위를 넘기지 않습니다.
- **무료·로컬 실행을 우선한다.** 사용하는 도구·라이브러리는 **무료(오픈소스 또는 무료 등급)** 를 기본으로 하고, **실습용 노트북에서 로컬로 실행 가능한** 가벼운 구성을 씁니다. 유료 디자인 도구·유료 호스팅·고사양 환경이 전제되는 선택지는 피하고, 꼭 필요한 경우에만 그 이유와 함께 **무료·로컬 대안**을 같이 제시합니다.
- **기대치를 맞춘다.** 본 구현과의 품질 차이, 다음 단계로 넘길 항목을 명확히 합의·기록합니다.

### Phase 2 — 와이어프레임 설계

**컴포넌트 목록**
- 공통: `Header`, `Sidebar`, `Button`(Primary/Secondary/Danger), `Input`(Text/Email/Password/Select), `Modal`, `Table`, `Card`
- 화면별: `LoginForm`, `DashboardShell`, `StatsCard` 등

### Phase 3 — 프로토타입 구현

- **합의 기록**: 승인된 화면, 수정 필요 항목, 추가 요구, 다음 단계 인계 사항

### 6.4 간이 상태 관리 (인증 예시)

```tsx
// src/hooks/useAuth.ts
import { useState } from "react";
import type { User } from "../types";

export function useAuth() {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(false);

  async function login(email: string) {
    setLoading(true);
    await new Promise((r) => setTimeout(r, 300)); // 모의 인증
    setUser({ id: 1, email, name: "테스트 사용자" });
    setLoading(false);
  }

  function logout() {
    setUser(null);
  }

  return { user, loading, login, logout };
}
```

- 문서에 없어 가정한 부분은 "(가정)"으로 표시해 데모에서 검증합니다.
- **바이브코딩 실습용 미니 앱**을 전제로 화면을 2~3개 수준으로 좁히고, MVP는 한 번의 실습으로 끝낼 수 있게 잡습니다. "클라이언트 합의"는 실습에서 스스로 점검 + 보여주기로 대신합니다.
- 도구·라이브러리는 **무료(오픈소스·무료 등급)** 로 갖추고, **실습용 노트북에서 로컬 실행**되는지 확인합니다. 유료·고사양이 불가피하면 그 이유와 무료·로컬 대안을 함께 적습니다.
