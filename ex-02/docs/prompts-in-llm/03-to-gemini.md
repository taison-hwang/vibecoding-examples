# 요구사항정의서 비판적 리뷰

## 🔴 사실 오류 / 수정 필요

### 1. Gemini API의 오디오 직접 입력 지원 (2단계)
> *"Gemini API의 멀티모달 기능(Audio 직접 입력 가능)을 활용하여 텍스트 변환 없이 바로 요약"*

**현재 사실:** Gemini 1.5/2.x는 오디오 파일을 직접 입력으로 받을 수 있습니다(File API 경유). 단, **브라우저 실시간 스트리밍 오디오**를 API로 직접 넘기는 것은 지원되지 않습니다. 브라우저 녹음 → 파일로 저장 → 백엔드에서 File API 업로드 → Gemini 처리 순서가 필요합니다. "텍스트 변환 없이 바로"라는 표현이 과도하게 단순화되어 있습니다.

**수정 제안:** "오디오 파일을 서버에 업로드 후 Gemini File API를 통해 처리"로 명시

---

### 2. Gmail SMTP + 앱 비밀번호 방식의 제약
> *"앱 비밀번호(App Password)를 발급받아 로컬 .env 환경변수에 세팅"*

**현재 사실:** Gmail 앱 비밀번호는 **Google 계정에 2단계 인증이 활성화된 경우에만 발급 가능**합니다. 또한 Google은 2024년부터 일부 계정 유형에서 앱 비밀번호 지원을 점진적으로 축소 중입니다. 실습 환경에서 "추가 비용 없이 바로 된다"는 전제가 무너질 수 있습니다.

**수정 제안:** 전제 조건(2단계 인증 활성화 필수)을 명시하거나, 대안으로 Mailtrap, Resend 등 개발용 SMTP 서비스를 병기

---

### 3. `gemini-2.5-flash` 무료 한도 표현
> *"무료 한도 넉넉함"*

**현재 사실:** 무료 등급(Free Tier)의 RPM(분당 요청 수), TPM(분당 토큰 수) 제한은 Google 정책에 따라 수시로 변경됩니다. MVP 실습 용도로는 충분하지만, "넉넉함"이라는 표현은 주관적이고 문서화 시점 이후 변경될 수 있습니다.

**수정 제안:** "실습 규모에서는 무료 한도 내 동작 예상 (정확한 한도는 Google AI Studio 공식 문서 확인 필요)"으로 완화

---

## 🟡 설계 공백 / 보완 필요

### 4. API 키 노출 위험 — 가장 중요한 보안 이슈
문서에 `.env` 사용을 언급하고 있으나, **프론트엔드(React)에서 Gemini API를 직접 호출하는 구조인지, 백엔드를 통해 호출하는 구조인지**가 명확하지 않습니다.

만약 Vite 환경에서 `VITE_GEMINI_API_KEY`로 프론트에 노출하면 **브라우저 소스에서 키가 그대로 노출**됩니다. 문서는 백엔드(`/api/summarize`)를 두는 구조를 제시하고 있으나, 이 점을 명시적 주의사항으로 강조해야 합니다.

**수정 제안:** "API 키는 반드시 서버 사이드(.env)에서만 사용, 프론트엔드에 절대 노출 금지" 항목 명시

---

### 5. Nodemailer는 Node.js 전용 — 기술 스택 혼재
> *"백엔드: Node.js(Express) **또는** Python(FastAPI)"*  
> *"이메일 전송: Nodemailer(Node.js) **또는** smtplib(Python)"*

선택지는 올바르게 병기되어 있으나, **두 백엔드 선택지가 독립적으로 완결되는 조합**임을 명시하지 않아 혼용 오해 가능성이 있습니다.

**수정 제안:** 표로 정리

| 백엔드 선택 | 이메일 라이브러리 |
|---|---|
| Node.js (Express) | Nodemailer |
| Python (FastAPI) | smtplib 또는 python-dotenv + smtplib |

---

### 6. 이메일 전송 실패 시 UX 처리 미정의

"완료 알림(Alert)"만 언급되어 있고, **전송 실패(네트워크 오류, SMTP 인증 실패, 잘못된 수신자 주소 등)에 대한 에러 처리**가 요구사항에 없습니다. MVP라도 실패 케이스 UX는 정의가 필요합니다.

**수정 제안:** 전송 실패 시 에러 메시지 표시 및 재시도 방법 명시

---

### 7. Gemini 응답 포맷 계약 부재

`POST /api/summarize`가 "요약 및 액션 아이템 JSON 반환"이라고 명시되어 있으나, **JSON 스키마가 정의되어 있지 않습니다.** Gemini는 프롬프트 설계에 따라 응답 형태가 달라지므로, 프론트엔드와 백엔드 간 계약이 없으면 파싱 로직 구현 시 혼란이 생깁니다.

**수정 제안:** 예시 응답 스키마 추가
```json
{
  "summary": "회의 핵심 개요 텍스트",
  "action_items": [
    { "owner": "홍길동", "task": "API 명세 작성", "due": "2025-07-01" }
  ]
}
```
