Claude의 비판적 리뷰는 실습 프로젝트의 성공과 시행착오 방지를 위한 매우 날카롭고 유용한 지적들입니다.

지적된 6가지 개선 사항(용어 정립, 모델 표현 수정, `.gitignore` 보안 추가, CORS 설정 누락 보완, Mailtrap 1순위 조정, 평가 매트릭스 패스/페일 조건 명시)을 모두 반영하여, 논리적으로 훨씬 정교하고 실용적으로 개선된 ADR을 작성했습니다.

# [ADR-0001] 안전한 Gemini API 호출 및 이메일 발송을 위한 2-Tier 클라이언트-서버 아키텍처 채택

## 상태

**승인됨 (Approved)**

## 배경

* **현재의 과제:** 스크럼 회의 직후 회의록 작성 및 액션 아이템 도출/공유에 따른 팀원들의 리소스 소모 및 피로감 가중. 논의된 업무의 휘발 방지 필요.
* **프로젝트의 목적:** 자연어 기반 스크럼 회의 데이터를 Gemini API로 구조화(JSON)하고, 이를 팀원들에게 이메일로 발송하는 바이브코딩 실습용 MVP 미니 웹앱 구축.
* **핵심 아키텍처 드라이버:**
* **크리티컬 보안 (제약 사항):** Gemini API 키가 프론트엔드(브라우저) 코드나 환경변수에 노출되는 것을 절대적으로 방지해야 함. (보안 요건 미충족 시 아키텍처 후보에서 즉시 탈락)
* **비용 및 환경 (제약 사항):** 실습용 노트북 로컬 환경에서 구동 가능해야 하며, 무료(오픈소스 및 무료 등급) 인프라/라이브러리를 활용해야 함.
* **단순성 및 신속성:** 회원가입이나 복잡한 DB 구축 없이 2~3개 화면 규모로 빠르게 동작하는 구조(MVP)여야 함.



## 결정

프론트엔드와 백엔드가 명확히 분리 실행되는 2-Tier 클라이언트-서버 아키텍처 (프론트-백 분리형 SPA + API 서버)를 채택합니다.

* **프론트엔드:** React, Vite, TypeScript, Tailwind CSS를 사용하여 단일 페이지 앱(SPA)으로 구축하며, 팀원 데이터는 브라우저 내 `LocalStorage`에 영속화합니다.
* **백엔드 API:** 프론트엔드의 요청을 받아 Gemini API 연동 및 이메일 발송을 대행하는 경량 API 서버(Node.js Express 또는 Python FastAPI)를 구축합니다.
* **주요 데이터 흐름:**
1. **요약:** 프론트엔드(회의록 입력) ➔ 백엔드(`/api/summarize`) ➔ Gemini API 호스팅(키 검증 및 요약 JSON 획득) ➔ 백엔드 ➔ 프론트엔드(결정된 JSON 시각화).
2. **이메일 발송:** 프론트엔드(발송 요청) ➔ 백엔드(`/api/send-email`) ➔ SMTP 서버(Mailtrap 또는 Gmail) ➔ 팀원 메일함.



## 근거

바이브코딩 지침(무료, 로컬 실행, 최소한의 복잡도)에 따른 후보 패턴 평가 결과는 다음과 같습니다. 크리티컬 드라이버인 **'보안(API 키 보호)' 항목에서 3점 미만(과락)을 받은 후보는 종합 점수와 관계없이 제외**합니다.

### 아키텍처 패턴 평가 매트릭스

| 평가 기준 | 후보 1: 클라이언트 단독 앱 (Serverless) | 후보 2: 2-Tier 클라이언트-서버 (선택) | 후보 3: 서비스 기반 / MSA |
| --- | --- | --- | --- |
| **보안 (API 키 보호)** `[Pass/Fail]` | 1 (**Fail**) | **5 (Pass)** | 5 (Pass) |
| **개발 공수·시간 효율** | 5 | **4** | 1 |
| **비용 (무료 등급 지원)** | 4 | **5** | 2 |
| **노트북 로컬 실행성** | 3 | **5** | 1 |
| **결과 및 종합 점수** | **탈락 (보안 과락)** | **19 / 20 (최종 채택)** | 9 / 20 |

* **선택 이유:** 후보 1(클라이언트 단독 구조)은 개발이 가장 빠르지만 브라우저에 API 키가 노출되므로 보안 기준 미달로 즉시 탈락했습니다. 따라서 로컬 가동이 쉬우면서 보안 환경변수(`.env`) 관리가 용이한 후보 2(2-Tier 클라이언트-서버)가 실습에 가장 안전하고 적합합니다.

## 대안

* **대안 1: 프론트엔드 단독 앱 (클라이언트에서 직접 Gemini API 호출)**
* *탈락 이유:* 브라우저 개발자 도구나 소스코드 빌드 시 API 키가 탈취될 위험이 있어 크리티컬 보안 지침 위반으로 배제했습니다.


* **대안 2: MSA (Microservice Architecture)**
* *탈락 이유:* 요약 서비스와 이메일 서비스를 독립된 마이크로서비스로 분리하는 것은 단독 실습생의 규모를 넘어서는 과도한 설계(Overengineering)이므로 제외했습니다.



## 영향

* **CORS(Cross-Origin Resource Sharing) 설정 필수:** 프론트엔드(보통 `localhost:5173`)와 백엔드(보통 `localhost:3000` 또는 `8000`)의 오리진(Origin)이 서로 다르므로, 백엔드 애플리케이션 설정 시 반드시 프론트엔드 주소에 대한 CORS 허용 인프라(미들웨어) 설정이 추가되어야 합니다.
* **운영 및 실습 측면:** 실습생은 로컬 PC에서 프론트엔드(Vite)와 백엔드(Node/Python) 서버 총 2개의 프로세스를 동시에 띄워야 하므로, 각 서버의 포트 충돌 및 통신 엔드포인트 관리가 필요합니다.
* **향후 확장:** 2단계 요구사항인 오디오 파일(.mp3, .wav) 업로드 기능 도입 시, 백엔드 서버에 멀티파트(Multipart) 파일 스트림 처리 로직만 추가하면 되므로 유연하게 대응 가능합니다.

## 비기능 요구 대응

* **보안:** Gemini API 키는 백엔드의 `.env` 파일에 저장하고 기밀성을 유지합니다. 프론트엔드 정규식을 활용하여 1차적으로 잘못된 수신자 주소 진입을 차단합니다.
* **성능 및 UX:** Gemini API 및 이메일 발송은 외부 네트워크를 경유하므로 지연(Latency)이 발생할 수 있습니다. 이를 위해 UI 단에서 **로딩 스피너 제공 및 버튼 비활성화(Disabled)** 처리를 강제하여 연속 호출을 방지합니다.
* **예외 처리:** Gemini 파싱 에러 발생 시 `재시도` 버튼을 제공하고, SMTP 전송 실패 시 실패 원인을 경고 모달/토스트로 명확히 시각화하여 사용성을 보장합니다.

## 기술 스택

모든 기술 스택은 **무료 및 오픈소스** 기준이며, **일반 노트북에서 추가 인프라 없이 즉시 로컬 실행**이 가능합니다.

* **프론트엔드 (Client):** React, Vite, TypeScript, Tailwind CSS *(로컬 실행 가능, 100% 무료)*
* **데이터 저장소:** 브라우저 내장 `LocalStorage` *(별도 RDB/NoSQL 설치 불필요)*
* **AI 모델:** `gemini-3.5-flash` *(Google AI Studio 무료 한도 활용, 멀티모달 텍스트 및 오디오 입력 지원)*
* **백엔드 및 이메일 조합 (실습 환경에 따라 택 1):**
* **조합 A (Node.js 환경):** Express(`cors` 미들웨어 필수 적용) + `Nodemailer` + `dotenv`
* **조합 B (Python 환경):** FastAPI(`CORSMiddleware` 필수 적용) + `smtplib` / `email.mime` + `python-dotenv`


* **SMTP 외부 연동 (실용성 최적화 순서):**
1. **Mailtrap / Resend (1순위 권장):** 개발자용 무료 등급을 지원하며, 복잡한 인증 절차 없이 로컬 환경에서 가장 빠르고 안정적으로 이메일 발송 테스트가 가능함.
2. **Gmail SMTP (2순위 선택):** 구글 계정의 2단계 인증 활성화 및 '앱 비밀번호(App Password)' 생성이 필요하므로 보안 설정 진입장벽이 존재함.
