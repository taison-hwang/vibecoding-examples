# [ADR-0001] 단일 페이지 애플리케이션(SPA)과 BFF 패턴 기반의 No-DB 클라이언트-서버 아키텍처 채택

## 상태

승인됨

## 배경

* **본 프로젝트의 목적**: 바이브코딩 교육을 위한 소규모 학습용 미니 프로젝트로, 매일 진행되는 스크럼 회의 내용을 요약하고 이메일로 발송하는 단일 페이지 웹 애플리케이션을 신속하게 구현하는 것입니다.
* **핵심 요구사항 및 제약**:
* 사용자로부터 긴 대화체 텍스트를 입력받아 Gemini API를 통해 마크다운 형식으로 요약해야 함.
* 도출된 마크다운 결과물을 단일 화면(SPA) 내 우측 결과창에 즉시 렌더링해야 함.
* 환경변수(`.env`) 관리를 통해 Gemini API Key와 Gmail 앱 비밀번호를 클라이언트 브라우저에 노출하지 않고 안전하게 취급해야 함.
* 학습 목적으로 구조를 단순화하기 위해 데이터베이스(DB)를 사용하지 않고 메모리와 프론트엔드 상태(State)로만 데이터를 유지해야 함.
* 단기 개발 효율성이 높아야 하며, 풀스택 흐름(AI API 연동, SMTP 연동)을 직관적으로 파악할 수 있어야 함.



## 결정

* **선택한 아키텍처 패턴**: 싱글 페이지 애플리케이션(SPA) 프론트엔드 + API 중계를 위한 **BFF(Backend for Frontend) 패턴 기반의 클라이언트-서버 구조**.
* **주요 구성 요소와 데이터 흐름**:
1. **Frontend (UI 레이어)**: React/Vite 기반 단일 레이아웃 화면. 이메일 형식 유효성 검증(Validation) 및 UI 로딩 상태 관리를 전담하며, 마크다운 뷰어를 통해 AI 요약본을 화면에 노출함.
2. **Backend (BFF 레이어)**: API 요청을 중계하는 프록시 역할 및 비즈니스 로직(서버 날짜 기준 이메일 제목 생성, 이메일 일괄 전송) 처리. `.env` 파일을 사용하여 외부 API 키를 로컬 개발 환경 수준에서 기본적으로 격리 관리함.
3. **External Services**: Gemini API(AI 추론), Gmail SMTP Server(이메일 발송).


* **데이터 규격 및 타이틀 규칙 확정**:
* **AI 출력 포맷**: 프론트엔드 마크다운 렌더링 동기화를 위해 반드시 `### 📌 스크럼 회의 요약`, `### 🎬 액션 아이템 (Action Items)`의 섹션 구조를 강제하는 프롬프트 엔지니어링을 적용함.
* **이메일 제목 템플릿**: `[스크럼 요약] YYYY-MM-DD 회의 공유` (서버 발송 시점 날짜 자동 매핑)



## 근거

* **용어 및 아키텍처의 정확성**: 본 구조는 여러 독립 서비스 간의 통신을 전제로 하는 엔터프라이즈 SOA(서비스 중심 아키텍처)보다, 프론트엔드를 지원하기 위해 프록시 및 가벼운 로직을 수행하는 **BFF 패턴의 클라이언트-서버 구조**로 정의하는 것이 학습 개념 확립에 정확함.
* **개발 공수 및 시간 효율성 극대화**: No-DB 제약 사항을 충족하여 영속성 레이어 구축 비용을 배제하고, 핵심인 'AI API 연동'과 '보안 인프라' 학습에 집중함.
* **보안 요구사항 만족**: 외부 API 키는 클라이언트(브라우저) 환경에 노출되면 안 되므로, 최소한의 백엔드 레이어를 두어 자격 증명을 보호함.

## 대안

* **대안 1: 완전한 서버리스 클라이언트 앱 (No Backend)**
* *탈락 이유*: Gemini API Key와 Gmail 앱 비밀번호가 브라우저 코드 내에 노출되거나 하드코딩될 위험이 매우 높아 환경변수 보안 관리라는 학습 목표에 위배됨.


* **대안 2: 관계형 데이터베이스(RDB)를 포함한 풀스택 구조**
* *탈락 이유*: 회의록 이력 저장 기능이 요구사항에 없고, "No DB"가 명시적 제약 조건이므로 오버 엔지니어링에 해당함.



## 영향

* **개발 및 운영**: 프론트엔드와 백엔드의 역할이 명확히 나뉘어 개발 효율성이 높음. 추후 2단계 확장 기능(브라우저 마이크 녹음기능, LocalStorage 활용한 이메일 목록 캐싱) 도입 시 백엔드 수정 없이 프론트엔드 확장만으로 구현 가능함.
* **보안 주의사항**: `.env` 방식은 로컬 개발 환경에서의 기본적인 격리 수단이므로, **`.gitignore` 파일에 `.env`를 반드시 포함**시켜 원격 저장소(Git)에 유출되지 않도록 강하게 통제해야 함을 학습 과정에서 강조함.
* **예외 및 실패 처리 방침**: 다수의 수신자에게 메일을 발송할 때 일부 주소가 잘못된 경우(부분 실패 케이스), MVP 수준의 학습 목적상 **복잡한 부분 예외 처리는 생략하고 전체 실패로 간주**하여 사용자에게 에러 알림(`alert`)을 노출하는 것으로 단순화함.

## 비기능 요구 대응

* **보안**: 구글 계정 2단계 인증 및 16자리 '앱 비밀번호(App Password)' 사용 필수.
* **성공/실패 피드백**: 프론트엔드단에서 1차 정규식 이메일 검증(쉼표 분리 검사)을 처리함으로써 불필요한 백엔드 요청 낭비 방지.

## 기술 스택

* **Frontend**: React.js / Vite + Tailwind CSS (UI 레이아웃 및 상태 관리)
* **Backend**: Node.js (Express 프레임워크)
* **Language**: JavaScript / TypeScript
* **AI Engine**: Google Gemini API SDK
* **Email Protocol**: Nodemailer 라이브러리 (Gmail SMTP 연동)
